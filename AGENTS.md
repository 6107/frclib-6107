# AI Coding Agent Guidelines for frclib-6107

## Architecture Overview

This is an FRC (FIRST Robotics Competition) Python library providing reusable subsystems, commands, and utilities built on **WPILib Commands-v2**. The core design centers on:

- **Pykit-based Logging**: All telemetry flows through `Logger` → AdvantageScope for real-time monitoring and replay analysis. DO NOT use `SmartDashboard` directly for data logging.
- **Constants-Driven Configuration**: Robot behavior is configured through `RobotConstants`, `SimulationConstants`, and `NetworkConstants` dataclasses (marked with `slots=True` for performance). Teams override these in their robot project.
- **Three Robot Modes**: REAL (on-field), SIMULATION (desktop), REPLAY (log playback with `LOG_PATH` env var). Logic must handle all modes appropriately.

## Code Entry Points & Structure

- **`src/lib_6107/robot.py`**: Base `Robot` class extends `LoggedRobot` with built-in pykit initialization, periodic tracking, and fault detection scaffolding. Always call `container_init()` callback in `robotInit()` so teams can provide custom subsystems.
- **`src/lib_6107/robotcontainer.py`**: Where subsystems are instantiated and commands bound to controllers. Teams subclass this to add their mechanisms.
- **`src/lib_6107/subsystems/`**: Base `SubsystemBase` class requires `_initialized = False` at start, set to `True` only after full `__init__` completes (prevents pykit race conditions). Implement optional hooks: `record_metadata()`, `fault_detection()`, `stop()`, `sim_init()`, `update_sim()`.
- **`src/lib_6107/commands/`**: Base `BaseCommand` class with standardized logging. Commands auto-add subsystem requirements via `addRequirements()`.

## Python Code Standards and Best Practices

When generating any Python code for this library, adhere to the following guidelines:
1 **PEP 8 Compliance*: Follow PEP 8 style guidelines for formatting, naming, and structure. Use tools like `black` and `flake8` for automated checks.
2 **Type Hints**: Use Python type hints for all function signatures and class attributes to improve readability and enable static analysis.
3 **Docstrings**: Provide clear docstrings for all classes, methods, and functions using the Google style format. Include descriptions of parameters, return values, and any exceptions raised.
4 **Logging**: Use the `Logger` class from `pykit` for all telemetry and debugging output. Avoid using `print()` statements or `SmartDashboard` for logging
5 **Performance**: Ensure that all code is optimized for performance, especially in periodic methods. Avoid blocking calls and ensure that any I/O operations are asynchronous or non-blocking.
6 **Testing**: Write unit tests for all new functionality using `pytest`. Ensure that tests are comprehensive and cover edge cases. Use mocks for hardware interactions where necessary.
7 **Documentation**: Update this `AGENTS.md` file with any new patterns, workflows, or architectural changes introduced by your code. Include examples where helpful. 
8 **Secuity Best Practices**: Follow security best practices, especially when handling network communication or external libraries. Use `bandit` for security analysis and address any issues it raises.
9 **Consistent Naming Conventions**: Use consistent naming conventions for classes, methods, and variables. For example, use `CamelCase` for classes and `snake_case` for functions and variables.
10 **Small Functions**: Keep functions small and focused on a single task. This improves readability and maintainability.
11 **Single Responsibility Principle**: Ensure that each class and function has a single responsibility. This makes the code easier to test and maintain.
12 **Error Handling**: Implement robust error handling, especially for hardware interactions. Use try-except blocks where appropriate and log any exceptions using the `Logger`.
13 **Imports**: Organize imports according to PEP 8 guidelines: standard library imports first, followed by third-party imports, and then local application imports. Use absolute imports where possible.
12 **Avoid Global State**: Minimize the use of global variables. Pass necessary state through function parameters or class attributes to improve modularity and testability.
13 **Use of Constants**: Define constants in the appropriate dataclasses (`RobotConstants`, `SimulationConstants`, `NetworkConstants`) and avoid hardcoding values throughout the codebase. This improves maintainability and readability.
14 **Documentation of Constants**: Provide clear documentation for each constant defined in the dataclasses, including its purpose and any relevant units or constraints.
15 **`@ai_generated` Decorator**: Use the `@ai_generated` decorator for any code generated by AI agents to clearly indicate its origin and facilitate future maintenance.
16 **Code Reviews**: All AI-generated code should be reviewed by a human developer to ensure it meets the above standards and integrates well with the existing codebase. Address any issues or improvements identified during the review process.
17 **Avoid Magic Numbers**: Replace magic numbers with named constants defined in the appropriate dataclass. This improves code readability and maintainability.

### Subsystem Initialization Pattern

```python
class MySubsystem(SubsystemBase):
    def __init__(self, container):
        super().__init__(container, "MySubsystem", "my/subsystem")
        # ... device init, config, etc ...
        self._initialized = True  # LAST line - prevents pykit race condition
```

### Simulation Support

Subsystems implementing simulation must:

1. Override `sim_init(physics_controller)` to store the controller
2. Override `update_sim(now, tm_diff)` to update WPILib simulation (called at ≥10 mS intervals)
3. `physics.py` defines overall mechanism simulation; subsystem `update_sim()` feeds back motor currents

## Logging & Telemetry Architecture

- **Logger.start()**: Called once in `Robot.__init__()` after mode detection
- **Data Receivers**:
  - **REAL**: NT4Publisher (NetworkTables) + WPILOGWriter (USB drive `/U/logs/` or fallback `/home/lvuser/pyLogs/`)
  - **SIMULATION**: WPILOGWriter + NT4Publisher
  - **REPLAY**: WPILOGReader (input) → WPILOGWriter (output `.wpilog` files)
- **LogTracer**: Tracks periodic performance; call `LogTracer.record()` for phase timings (see `robotPeriodic()`)

**Never disable auto-logging manually on real robots** — that's handled in `robotInit()` for Phoenix6 and REV StatusLogger.

## Performance Optimization (Critical for Competition)

- **Target Loop Time**: ≤20 mS per periodic cycle
- **Don't Block**: All I/O (CAN, vision, sensors) must be asynchronous/non-blocking
- **Profile with LogTracer**: Use `resetOuter()`, `record()`, `recordTotal()` to identify bottlenecks
- **RobotStatistics**: Tracks periodic duration moving average; visible in SmartDashboard for tuning

## Motor Vendor Integration

### CTRE Phoenix6 (TalonFX primarily)

- Imported: `from phoenix6.hardware import TalonFX`
- Auto-logging disabled in `robotInit()` via `SignalLogger.enable_auto_logging(False)`
- Telemetry: Use Logger/pykit, not Phoenix's built-in logging

### REV Robotics (SparkMax/SparkFlex)

- Imported: `from rev import SparkMax, SparkFlex, SparkMaxSim, SparkFlexSim, SparkClosedLoopController`
- Auto-logging disabled: `StatusLogger.disableAutoLogging()`
- Sim support: SparkMaxSim/SparkFlexSim for simulation mode
- Config: Use SparkBaseConfig with proper error checking

## Vision & PathPlanning

- **PhotonVision**: `photonlibpy` library. DriveSubsystem often has tag tracking via `VisionSubsystem`
- **PathPlanner**: `LocalADStar` pathfinder set globally in `robotInit()`. Override in custom containers.
- **Limelight**: Support via `limelightlib-python` (3D pose estimation compatible)

## Common Workflows When Modifying This Library

1. **Adding a Subsystem Base Feature**: Extend `SubsystemBase`, mark `_initialized` pattern, add optional hooks.
2. **Modifying Constants**: Edit `RobotConstants` dataclass slots, ensure properties work (see `THETA_CONTROLLER_CONSTRAINTS()`).
3. **New Command Pattern**: Start from `BaseCommand`, add pykit logging, test in simulation mode first.
4. **Performance Regression**: Use `LogTracer` to profile, check motor update frequency and logging throughput.
5. **Simulation Fidelity**: Update `physics.py` with mechanism dynamics, use `BatterySim` for overall power draw.
